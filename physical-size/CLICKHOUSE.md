# ClickHouse Integration — `symphony.file_physical`

Companion to `DESIGN.md`. Covers how the walker's output becomes a ClickHouse table and how panels consume it. All queries shown here target the existing server: ClickHouse **26.3.9.8** on `localhost:8123`, user `symphony`, db `symphony`.

---

## 1. Why a second table (and not a column)

- `symphony.scan_results` is **populated by Symphony AdminCenter** via its JDBC exporter — the schema is auto-generated by AdminCenter's "CREATE TABLE recommendations for ClickHouse" (admin guide §G.3.3). We don't own it. Adding a column means the next time Symphony re-creates or validates the schema, we could get a mismatch.
- The only size column Symphony exports is `size` (logical). Confirmed: `DESCRIBE symphony.scan_results` shows one `size UInt64`, no `allocated`/`physical`/`on_disk` field. Confirmed in admin guide: §G.3.2 is terse ("Scan Policies may be configured to export individual file metadata"), and nowhere is an allocated/physical/compressed-size field mentioned.
- So we write to a **sibling table we own**, joined at query time. Symphony stays pristine.

## 2. Schema

```sql
CREATE TABLE IF NOT EXISTS symphony.file_physical (
    run_id     String,
    file_uri   String,
    allocated  UInt64,
    status     LowCardinality(String),  -- OK | MISSING | ERROR
    probed_at  DateTime64(3) DEFAULT now64(3)
)
ENGINE = ReplacingMergeTree(probed_at)
PARTITION BY run_id
ORDER BY (run_id, file_uri);
```

Choices, and why:

- **`ReplacingMergeTree(probed_at)`** — lets us re-walk a `run_id` without having to clear the table first. On merge, the latest `probed_at` wins per `(run_id, file_uri)`. At read time, pairs with `FINAL` or `argMax` to get authoritative rows (see §5).
- **`PARTITION BY run_id`** — makes "drop an old run" a one-line `ALTER TABLE DROP PARTITION`. With 9.96 M rows per run, each partition is still a healthy size for MergeTree.
- **`ORDER BY (run_id, file_uri)`** — mirrors `scan_results`' order so `(run_id, file_uri)` joins are efficient on both sides. Sparse primary index will skip straight to the right granule.
- **`LowCardinality(String)`** for `status` — three values, reduces storage and speeds up filters.
- **No codec tuning** — default LZ4 is fine at this size. Revisit if we ever hold many runs.

## 3. Load pipeline

Three stages, all stream-oriented so we never materialise 10 M rows in memory:

```
[ ClickHouse SELECT ]  --TSV-->  [ probe-physical-size.ps1 ]  --TSV-->  [ ClickHouse INSERT ]
```

### 3.1 Selecting the input

```sql
SELECT run_id, file_uri
FROM symphony.scan_results
WHERE run_id = '{RUN_ID}'
  AND filename != ''           -- skip directory entries (2,692 rows in this run)
FORMAT TabSeparated
```

- **`filename != ''`** is essential: 2,692 rows in `scan_results` are directory entries (trailing `/`, empty `filename`, `size = 0`). `GetCompressedFileSizeW` on a directory is undefined behaviour for our purposes, and the allocation number doesn't mean anything comparable. We exclude them.
- After filter: 9,962,001 rows to probe.

### 3.2 The walker (PowerShell 7, detailed in `DESIGN.md` §3.2)

Reads TSV on stdin, emits `run_id \t file_uri \t allocated \t status` on stdout. One-shot per file via P/Invoke — no intermediate files.

### 3.3 Inserting into ClickHouse

```bash
clickhouse-client \
  --user symphony --password symphony \
  --query "INSERT INTO symphony.file_physical (run_id, file_uri, allocated, status) FORMAT TabSeparated"
```

`clickhouse-client` streams stdin directly into batched INSERTs; we don't need to tune `max_insert_block_size` for our volume (10 M rows inserts in one or two parts).

### 3.4 Full pipeline

The `load-physical-size.sh` driver will pipe all three stages so no intermediate file is required. Optional `tee` to `physical_sizes.tsv` preserves a local artifact for replay without re-walking.

```bash
RUN_ID="c915d505-3f5b-4bae-963b-c521b7fd63e3-1776679196233"

curl -s "http://localhost:8123/?user=symphony&password=symphony&default_format=TabSeparated" \
  --data-binary "SELECT run_id, file_uri FROM symphony.scan_results WHERE run_id='$RUN_ID' AND filename != ''" \
| pwsh -NoProfile -File ./probe-physical-size.ps1 \
| tee physical_sizes.tsv \
| curl -s "http://localhost:8123/?user=symphony&password=symphony&query=INSERT%20INTO%20symphony.file_physical%20FORMAT%20TabSeparated" \
    --data-binary @-
```

(`clickhouse-client` would be cleaner; curl works even if the client isn't installed on the Windows host.)

## 4. Throughput / resource budget

- Volume: 9,962,001 files to probe, each a metadata-only Win32 call on NVMe.
- Prior-art per-call cost: low tens of µs for `GetCompressedFileSizeW`.
- Walker target: **< 30 min** single-threaded, **< 10 min** at `-ThrottleLimit 8`. If we miss that, escalate to a small .NET tool (same API, lower overhead).
- ClickHouse insert: trivial at this row count on this machine. No special tuning.
- Storage: ~9.96 M rows × ~100 bytes raw ≈ **~1 GB** uncompressed; probably **~100-200 MB on disk** after LZ4 + LowCardinality compression.
- WSL keepalive **must** be running throughout (per `project_symphony_clickhouse` memory). Walker should ping `http://localhost:8123/ping` before starting and abort with a clear message if ClickHouse doesn't respond — otherwise a mid-run VM idle-shutdown corrupts the stream.

## 5. Read-side patterns (how panels consume it)

ClickHouse has two reasonable patterns here. `DESIGN.md` showed the join; this section is the decision rationale.

### 5.1 Option A — LEFT JOIN (simple, portable)

```sql
SELECT
  sum(
    if('$size_metric' = 'Physical', fp.allocated, sr.size)
  ) AS bytes
FROM symphony.scan_results sr
LEFT ANY JOIN (
  SELECT run_id, file_uri, argMax(allocated, probed_at) AS allocated
  FROM symphony.file_physical
  WHERE run_id = '$run_id' AND status = 'OK'
  GROUP BY run_id, file_uri
) fp
  ON fp.run_id = sr.run_id AND fp.file_uri = sr.file_uri
WHERE sr.run_id = '$run_id'
```

- `LEFT ANY JOIN` = one row per left row even if the right side has duplicates (defensive against `ReplacingMergeTree` merges in flight).
- The subquery with `argMax` handles re-probes deterministically without relying on `FINAL`.
- `if('$size_metric' = 'Physical', ...)` means the same query shape drives both Logical and Physical tiles.
- Degrades cleanly: if `file_physical` is empty for this `run_id`, `fp.allocated` is `NULL` → `if(..., NULL, sr.size)` — still returns logical when Physical is selected but no data exists. If that's not the behaviour we want we can `ifNull(fp.allocated, sr.size)` or similar; revisit once Grafana wiring is live.
- **Good for Grafana** because every panel is already a standalone SELECT. Simple substitution.

### 5.2 Option B — Dictionary lookup

```sql
CREATE DICTIONARY symphony.dict_file_physical (
    run_id     String,
    file_uri   String,
    allocated  UInt64
)
PRIMARY KEY run_id, file_uri
SOURCE(CLICKHOUSE(
    QUERY 'SELECT run_id, file_uri, argMax(allocated, probed_at) AS allocated
           FROM symphony.file_physical
           WHERE status = ''OK''
           GROUP BY run_id, file_uri'
))
LIFETIME(MIN 300 MAX 600)
LAYOUT(COMPLEX_KEY_HASHED());
```

Panels then use:

```sql
SELECT
  sum(
    if('$size_metric' = 'Physical',
       dictGetUInt64('symphony.dict_file_physical', 'allocated', (sr.run_id, sr.file_uri)),
       sr.size)
  ) AS bytes
FROM symphony.scan_results sr
WHERE sr.run_id = '$run_id'
```

- Dictionary is **resident in RAM** on the ClickHouse server — ~10 M rows × (string-key + UInt64 value) ≈ **~1.5 GB RAM** with `COMPLEX_KEY_HASHED`. Non-trivial on a laptop-class demo box.
- Lookup is O(1) amortized. On a simple aggregate scan across 10 M rows this beats the join.
- Cost: the dictionary takes a few seconds to refresh after each walker run (or however `LIFETIME` is set).

### 5.3 Which to use

**Start with Option A (LEFT JOIN).** Reasons:
- Zero server-side state to maintain. Every panel is self-contained.
- ClickHouse 26 joins on sorted primary keys are very fast; the `(run_id, file_uri)` prefix on both sides means the planner streams both tables in order.
- Memory cost is bounded per query rather than a standing RAM tax.

Escalate to Option B only if actual panel latency in Grafana becomes a problem at demo time.

## 6. Re-probe semantics

- Re-running the walker against the same `run_id` **appends** new rows.
- `ReplacingMergeTree(probed_at)` collapses duplicates on background merge; reads via `argMax(...)` are deterministic whether or not merge has run.
- If we want to force a clean state for a `run_id` (e.g. walker had a bug): `ALTER TABLE symphony.file_physical DROP PARTITION '{RUN_ID}'` then re-walk.

## 7. Validation queries (saved as `validation.sql`)

```sql
-- Coverage: should equal scan_results count (excluding directories)
SELECT
  (SELECT count() FROM symphony.scan_results WHERE run_id='$run_id' AND filename != '') AS expected,
  (SELECT uniqExact(file_uri) FROM symphony.file_physical WHERE run_id='$run_id')        AS probed,
  (SELECT countIf(status != 'OK') FROM symphony.file_physical WHERE run_id='$run_id')    AS non_ok;

-- Physical total — should land near the generator's ~1.2 TB
SELECT
  formatReadableSize(sum(allocated)) AS physical_total
FROM symphony.file_physical
WHERE run_id = '$run_id' AND status = 'OK';

-- Ratio check
SELECT
  formatReadableSize((SELECT sum(size) FROM symphony.scan_results WHERE run_id='$run_id')) AS logical,
  formatReadableSize((SELECT sum(allocated) FROM symphony.file_physical WHERE run_id='$run_id' AND status='OK')) AS physical,
  round(
    (SELECT sum(allocated) FROM symphony.file_physical WHERE run_id='$run_id' AND status='OK')
    / (SELECT sum(size) FROM symphony.scan_results WHERE run_id='$run_id' AND filename!=''),
    4
  ) AS physical_over_logical;

-- Distribution of physical size
SELECT
  formatReadableSize(allocated) AS bucket,
  count() AS files,
  formatReadableSize(sum(allocated)) AS bytes
FROM symphony.file_physical
WHERE run_id = '$run_id' AND status = 'OK'
GROUP BY allocated
ORDER BY allocated;
```
